1. 执行摘要

当前分支为 xLLM 的 MLU 后端引入 Triton JIT 运行时能力，用于替代部分 MLU kernel 的静态 AOT/TMO 预编译路径。新方案允许 C++ kernel wrapper 在运行时根据实际参数签名、constexpr、launch 配置和设备信息触发 Triton 编译，并将生成的 MLU 二进制产物加载后直接提交到当前 MLU 队列执行。

该能力的核心价值是降低 AOT 形状组合穷举成本，使 selected MLU kernel 能覆盖更多 head count、dimension、batch/sequence 组合，同时保留现有 C++ kernel API 对上层模型和 layer 的兼容性。

1 当前实现聚焦 MLU 后端，CUDA/NPU 后端接口预留但未实现。
2 C++ 本地缓存 + Triton 文件系统缓存共同降低重复编译成本。
3 支持 xLLM 内置 Triton kernel 与 torch_mlu_ops.triton.* kernel。
4 设计重点覆盖签名一致性、graph capture 安全、wheel 安装路径可解析。
关键结论： 该设计把 JIT 编译编排、参数描述协议和后端加载/launch 明确解耦。短期用于 MLU kernel 形状覆盖扩展，长期可作为 xLLM 多后端 Triton runtime 的统一抽象基础。

2. 目标与边界

2.1 功能目标

  • 允许选定 MLU kernel 根据真实运行参数懒编译 Triton specialization。
  • 支持 dotted module、pkg:// 和文件路径三类 kernel 源定位方式。
  • 保持现有 C++ kernel wrapper API 不变，减少对模型层的侵入。
  • 支持 std::optional<torch::Tensor> 和 nullptr，避免 nullable pointer 造成 runtime argument 偏移错误。

2.2 性能目标

  • 同一进程内，相同设备、签名和 launch 配置的 specialization 最多编译一次。
  • 热路径只保留参数打包、cache lookup 和 cnInvokeKernel。
  • 支持在非 graph capture 场景下进行可选 autotune。

2.3 非目标

  • 不承诺完全消除 cold-start latency。首次遇到新 specialization 时仍可能编译。
  • 不提供通用用户态 Python kernel API。
  • 不在本分支实现 CUDA/NPU 的 Triton JIT launch 后端。
  • 不替换全部 MLU kernel，只迁移当前选定路径。
  • 不实现跨进程编译产物管理，除 Triton 自身 cache 外不额外管理持久缓存。

3.3 设计原则

稳定协议

C++ 不解析手写字符串签名，而是向 Python 传递结构化参数描述。Python 通过 Triton 的 JITFunction.params 返回真实 tl.constexpr 和 do_not_specialize 信息。

后端隔离

JITKernel 和 triton_compile.py 负责通用编译编排，MluBackend 负责 MLU artifact 加载与 launch。

确定性缓存

缓存键覆盖参数类型、specialization hint、constexpr、launch 配置和 device，避免错误复用。

安装安全

runtime import path 从 build-tree 或 installed package 动态解析，不依赖构建机绝对路径。

4. 核心工作流

4.1 运行时生命周期

  1. 模型层调用现有 MLU kernel API。
  2. MLU wrapper 准备输入输出 tensor、shape、stride、constexpr 参数、Grid 和 LaunchCfg。
  3. wrapper 调用 JITKernel::get(py_path, fn_name) 获取 kernel 单例。
  4. 首次使用时，JITKernel 初始化 embedded Python，并调用 triton_compile.signature() 拉取 Python Triton 函数参数元信息。
  5. JITKernel::launch 按 Python 参数顺序处理 C++ 参数，生成 ArgPack 和 SpecList。
  6. 根据 SpecList + LaunchCfg + device 生成 specialization cache key。
  7. cache hit 时直接复用 CompiledKernel。
  8. cache miss 时通过 embedded Python 调用 triton_compile.compile()，构造 ASTSource 并调用 triton.compile。
  9. MluBackend 根据 Triton cache dir 创建 MluCompiledKernel。
  10. 首次 launch 时加载 .json metadata 和 .cnbin，校验 ISA，并获取 CN kernel。
  11. 最终通过 cnInvokeKernel 将 kernel 提交到当前 MLU queue。

4.4 Autotune 流程

Autotune 是可选能力。调用方可以通过 JITKernel::set_autotune(true, space) 启用。如果未传入搜索空间，后端提供保守默认值。autotune 的 winner 按 (signature, device) 缓存，不包含 LaunchCfg，后续同签名 launch 可自动使用最优配置。

Graph capture 注意事项： 如果当前 stream 处于 graph capture，autotune 会跳过并输出 warning。生产环境应在进入 capture 前完成必要 specialization 的 warmup。

5. 关键数据结构与接口

5.1 Launch 类型

struct Grid {
  uint32_t x = 1;
  uint32_t y = 1;
  uint32_t z = 1;
};

struct LaunchCfg {
  int32_t num_warps = 1;
  int32_t num_stages = 1;
};

Grid 表示 Triton program grid；LaunchCfg 表示会影响编译和 launch 的 Triton meta 参数。MLU 后端会将其转换为 CN runtime 的 grid 与 kernel class。

5.2 参数描述

enum class Kind {
  PTR,
  VAL,
  CONST,
};

enum class SpecHint {
  NONE,
  DIV16,
  EQ1,
};

struct ArgSpec {
  Kind kind = Kind::VAL;
  std::string type;
  SpecHint hint = SpecHint::NONE;
  bool specialize = true;
  std::string const_val;
};

using SpecList = std::vector<ArgSpec>;
  • PTR 用于 tensor、optional tensor 和 null pointer。
  • VAL 用于 runtime 标量。
  • CONST 用于 Python 侧 tl.constexpr 参数。
  • DIV16 和 EQ1 传递 Triton specialization hint。

5.3 JITKernel API

class JITKernel final {
 public:
  static JITKernel& get(std::string py_path, std::string fn_name);

  template <typename... Args>
  void launch(void* stream, Grid grid, LaunchCfg cfg, Args&&... args);

  void set_autotune(bool on, std::vector<LaunchCfg> space = {});

  template <typename... Args>
  void autotune(void* stream, Grid grid, Args&&... args);
};

JITKernel 是 C++ 调用侧核心接口。它将 py_path + fn_name 映射到进程内单例，并为该 kernel 维护 signature、compiled specialization cache 和 autotune winner。

5.4 Backend API

class CompiledKernel {
 public:
  virtual ~CompiledKernel() = default;
  virtual void launch(void* stream, Grid grid, const void* const* args) = 0;
};

class TritonBackend {
 public:
  virtual const char* name() const = 0;
  virtual int32_t current_device() const = 0;
  virtual std::string compile_options_json() const = 0;
  virtual std::unique_ptr<CompiledKernel> load_kernel(
      const std::string& cache_dir,
      const std::string& name) const = 0;
  virtual void sync(void* stream) const = 0;
  virtual bool is_capturing(void* stream) const;
};

TritonBackend 是后端差异的隔离点。当前 concrete backend 是 MluBackend，未来 CUDA/NPU 若要接入，应主要补齐 artifact loading、stream sync、graph capture 检测和 launch ABI。

5.5 Python 编译入口

def signature(py_path: str, fn_name: str) -> list[dict]:
    """返回每个参数的 is_constexpr / do_not_specialize 元数据。"""

def compile(
    py_path: str,
    fn_name: str,
    args_spec: list,
    backend: str,
    extra_opts: dict,
    num_warps: int,
    num_stages: int,
    device_id: int,
) -> str:
    """编译一个 specialization，并返回 Triton cache directory。"""

Python helper 兼容 Triton 3.1、3.2、3.3+ 的 AttrsDescriptor 与 ASTSource 差异，并将 MLU 编译选项合并到 num_warps、num_stages 基础配置上。

6. 编译与缓存设计

6.1 Specialization Key

serialize_key(specs, cfg, device) 覆盖所有影响生成代码或 launch 行为的字段：

  • 参数 kind
  • Triton base type
  • pointer marker
  • specialize flag
  • specialization hint
  • constexpr value
  • num_warps
  • num_stages
  • device id

6.2 两层缓存

C++ 进程内缓存

JITKernel::kernels_ 以 xLLM serialization key 作为键，避免同一进程内重复 Python 调用和重复创建 CompiledKernel。

Triton 文件系统缓存

Triton 根据 compile hash 管理 cache dir。Python helper 返回该目录，MLU 后端从中加载 .cnbin 和 .json。

6.3 冷热路径对比

路径 主要动作 性能影响
Cold path 初始化 Python、读取签名、构建 ASTSource、Triton 编译、加载 MLU artifact、launch。 可能带来显著首次调用延迟，应通过 warmup 控制。
Hot path 参数打包、cache lookup、复用 CompiledKernel、cnInvokeKernel。 接近手写 wrapper launch 开销，避免重复编译。

7. 构建与打包

7.1 CMake 集成

  • xllm/core/CMakeLists.txt 仅在 USE_MLU 下添加 triton_jit 子目录。
  • xllm/core/triton_jit/CMakeLists.txt 依赖 Python Development、pybind11::embed、nlohmann_json 和 MLU vendor libraries。
  • xllm/core/kernels/mlu/CMakeLists.txt 将 mlu_kernels 链接到 :triton_jit、torch_mlu 和 torch_mlu_ops。

7.2 Wheel staging

  • _stage_mlu_triton_kernels 将 xllm/core/kernels/mlu/triton_kernel/*.py 拷贝到 installed package。
  • _stage_triton_jit_scripts 将 triton_compile.py 拷贝到 xllm/core/triton_jit/scripts。
  • 必要的 __init__.py 会在 staging 阶段补齐，确保 embedded Python 可 import。

7.3 Import path 策略

prepare_triton_compile_import_path() 从当前 loaded object path 反推出 xLLM Python package root，支持 installed package、build-tree build/lib.* 和已加载 namespace package 场景。发现 root 后，它会更新 sys.path 和相关 package 的 __path__，保证 embedded Python 能解析编译脚本和 kernel 源文件。

8. 测试与验证

8.1 现有测试目标

新增测试位于 tests/core/kernels/mlu/，CMake target 为 triton_jit_ops_test。该 target 链接 :triton_jit、:mlu_kernels、torch、glog 和 GTest::gtest_main，并设置 PYTHONPATH 支持 build-tree import。

8.2 覆盖场景

  • fused_gdn_gating 第二次调用 cache hit，支持未预编译 head count。
  • causal_conv1d_update_decode 第二次调用 cache hit，支持未预编译 dimension。
  • fused_recurrent_gated_delta_rule_packed_decode 覆盖 L2 norm 开关和 state 更新一致性。
  • causal_conv1d_fn 覆盖 cache reuse 与未预编译 dimension。
  • ChunkGatedDeltaRule 覆盖 output_final_state=false 不写 final state 的回归。
  • recurrent gated delta rule 覆盖 beta shape 相关回归路径。

8.3 建议验收标准

维度 验收标准
正确性 重复调用输出在容差内一致；state/cache 原地更新符合预期；未编译 shape 输出 finite。
性能 第二次调用不触发 compile；warmup 后端到端延迟无异常抖动。
兼容性 build-tree 与 installed wheel 均能 import 编译脚本和 kernel modules。
Graph 模式 进入 capture 前完成 warmup；capture 内不触发 compile/autotune。
可观测性 首次 compile、autotune winner、失败候选和 CN runtime 错误具备可定位日志。

9. 发布与回滚

9.1 发布计划

  1. 保持特性仅在 USE_MLU 构建下启用。
  2. 仅对当前迁移的 MLU kernel wrapper 启用 JIT 路径。
  3. 在 MLU 硬件上执行 kernel-level regression tests。
  4. 对使用 causal conv 和 gated delta rule 的模型路径执行模型级 smoke test。
  5. 为已知模型 shape 与请求模式增加部署前 warmup。
  6. 上线后观察 first-token latency、cache miss 和 compile failure。

9.2 回滚策略

  • 将受影响 wrapper 回退到旧非 JIT 实现。
  • 临时禁用依赖 JIT-only kernel 的模型路径。
  • 生产部署回退到上一稳定分支或构建产物。

9.3 回滚触发条件

  • 支持 shape 中出现 Triton compile failure。
  • MLU artifact 与设备 ISA 不匹配。
  • 稳态服务中持续发生 cold compile。
  • graph capture 因未 warmup 或 JIT 行为失败。
  • stateful kernel 出现正确性回归。

10. 风险与未决问题

10.1 主要风险

风险 影响 缓解措施
冷启动编译延迟 首次请求或新 specialization 请求延迟升高。 上线前 warmup；服务指标区分 compile latency 与 steady-state latency。
C++ wrapper 与 Python kernel 签名漂移 参数偏移错误，可能导致 launch 崩溃或结果错误。 使用 signature() 元数据和参数数量断言；每个 wrapper 配套测试。
Triton 版本 API 变化 ASTSource 或 attrs 构造失败。 在 triton_compile.py 中隔离版本分支；固定并验证支持版本。
wheel 缺少 runtime Python 资产 installed package 下 embedded Python import 失败。 setup.py staging；增加 installed-wheel smoke test。
graph capture 内触发编译或 autotune capture 失败或性能异常。 capture 前 warmup；autotune 检测 capture 并跳过。
