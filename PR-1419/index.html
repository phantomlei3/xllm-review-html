xLLM Composite Block Manager 机制说明

当前分支前两个 commit 为后续 DeepSeek V4 的 composite block 机制打底：把原先“一个序列只对应一组 KV cache block”的假设扩展为“一个序列可以同时持有多个 block group”，并让分配、估算、驱逐、batch 构建、跨进程传输和测试都理解这个组合结构。

要解决的问题

旧假设：单一 block table

原 KV cache 路径把每个序列建模为一维 blocks_。一个 BlockManager 按统一 block_size 分配物理 block，batch 阶段输出一个 block_tables tensor。 

Sequence
  KVCacheState::blocks_ = [B0, B1, B2]

ForwardInput
  block_tables = [
    [B0, B1, B2]
  ]

新需求：多组 cache 同时存在

DeepSeek V4 后续需要不同形态的 KV/cache 资源并行管理：某些资源按固定 per-sequence ring 预留，某些资源按 token 增长。它们不能被塞进一个统一 block_size 的线性表里。 

Sequence
  composite_blocks_ = [
    group0: [R0, R1],        // RING, 固定块数
    group1: [T0, T1, T2],    // TOKEN, 按 token 增长
    group2: [D0]             // TOKEN, 另一种粒度
  ]

ForwardInput
  multi_block_tables = [
    group0_table, group1_table, group2_table
  ]
这个分支的核心不是“多分配几个 BlockManager”，而是把所有原来依赖单组 KV blocks 的调用点显式区分为：单组兼容路径、组合路径、以及二者共享的抽象接口。

兼容原则

单组路径仍保留 prefix cache、cache upload、beam search、raw block allocation 等能力。

隔离原则

组合路径禁止 prefix cache / cache upload / beam search / raw block API，防止旧语义误用组合态。

估算原则

调度不再按一个 block count 推断释放量，而是按 group 粒度累加 free + releasable。

组合模型

CompositeBlockPlan 是组合模式的静态规格。每个 group 都有自己的 block size、容量和行为类型。当前支持两类 group：RING 固定 per-sequence 块数，TOKEN 随 target tokens 增长。

Plan

group 0 RING

tokens_per_block=16，fixed_blocks_per_sequence=2。首次分配时一次性拿固定数量，后续不增长。

group 1 TOKEN

tokens_per_block=4。target tokens=9 时需要 ceil(9/4)=3 个 block。

group 2 TOKEN

tokens_per_block=10。target tokens=9 时需要 ceil(9/10)=1 个 block。

分配流程

组合模式下，BlockManagerPool 仍是外部入口，但不再直接向 block_managers_[dp_rank] 要 blocks。它先选 dp rank、分配 scheduler-side single block，然后把真正 KV block 分配委托给 CompositeSequenceBlockAllocator。

1

Scheduler

调用 allocate(sequence, target_num_tokens)。target 通常来自 prefill/decode 当前步预算。

2

select_dp_rank

对每个 allocator 调用 estimate_allocate，选择 bottleneck_free_sequences 最大的 dp rank。

3

single block id

分配 per-sequence 单块资源；失败则回滚新绑定的 dp_rank。

4

init composite state

首次分配时 resize composite_blocks_ 并写入 KVCacheGroupState。

5

new_block_counts

RING 首次固定分配；TOKEN 按 ceil(target/tokens_per_block) 补齐。

6

commit or rollback

所有 group 容量足够才逐组分配；中途失败释放新增 blocks 并恢复旧大小。

BlockManagerPool::allocate(sequence, num_tokens)
  dp_rank = select_dp_rank(sequence, num_tokens)
  allocate_single_block(sequence, dp_rank)

  if composite_block_plan.has_value():
    ok = allocators_[dp_rank]->allocate_sequence(sequence, num_tokens)
    if ok:
      return true
    rollback single block and newly bound dp_rank
    return false
CompositeSequenceBlockAllocator::new_block_counts(blocks, target)
  for each group:
    if group.kind == RING:
      if current_blocks == 0:
        new_counts[i] = group.fixed_blocks_per_sequence
      else:
        CHECK(current_blocks == fixed_blocks_per_sequence)

    if group.kind == TOKEN:
      target_blocks = ceil(target / group.tokens_per_block)
      new_counts[i] = max(0, target_blocks - current_blocks)
old_sizes = current composite_blocks_[i].size()
for each group:
  allocate extra blocks
  if allocation fails:
    for each group with new appended blocks:
      deallocate appended blocks
      resize back to old_sizes[i]
    if total composite blocks == 0:
      clear composite group metadata
    return false

调度与驱逐

旧驱逐判断

旧代码大体按单组 block 数估算：目标需要多少 block，就从 running queue 尾部累加可释放 blocks，够了就驱逐相应数量请求。

needed = ceil(target_tokens / block_size)
can_evict = sum(candidate.num_kv_blocks)
return can_evict >= needed

这个模型无法表达多 group，也容易在 prefix cache / shared block / dp rank 场景下高估释放量。

新驱逐判断

新代码把“释放后是否够用”的判断下沉到 KVCacheManager。它先收集候选 sequence，再按目标所在 dp rank 和 group 维度检查。

estimate = allocator.estimate_allocate(target, target_tokens)
add_release_capacity(allocator, candidates, estimate.groups)
return every group:
  free_blocks + releasable_blocks >= needed_blocks

组合路径按 group 累加；单组路径还会去重 shared block，并识别 prefix cache 持有的额外引用。

1

按 dp rank 隔离

已绑定目标只看同 rank candidate；未绑定目标逐 rank 试算，但不会把不同 rank 的释放量混加。

2

按 group 求和

组合路径的 release estimate 是每个 group 的 releasable_blocks，不再依赖空的 legacy kv_blocks。

3

精确删除队列元素

DecodePriorityQueue 新增 erase，驱逐不再假设被选中的请求一定是 back。

Batch 元数据链路

分配只是第一步。模型 forward 需要拿到各组 block id，所以这个分支把 multi_block_tables 从 BatchInputBuilder 贯穿到 ModelInputParams、RawForwardInput、proto 与 shared memory。

普通本地 forward

BatchInputBuilder::state_to_forward_input() 将每个 group 的二维 block table pad 成 tensor，放入 ModelInputParams::multi_block_tables。

分布式 / shared memory forward

RawForwardInput 保存原始三维 vector；proto 和 shared memory 都显式写入 manager_num，再逐 group 传 tensor。

边界与结论

当前明确不支持

组合模式下不支持 prefix cache、cache upload、beam search、raw block allocation、deallocate_without_cache、XTensor padding。代码用 capability 或 CHECK 防误用。

为什么要抽 allocator

如果 BlockManagerPool 继续直接操作 BlockManager，就必须在每个调用点手写组合分支。SequenceBlockAllocator 把差异集中在 sequence 语义层。

后续接 DeepSeek V4

后续模型侧可以消费 multi_block_tables，根据 group 类型和 block table 展开 attention/DSA 所需 metadata，而不再受单一 block table 形状限制。

一句话总结

这两个 commit 把 xLLM 的 KV cache 管理从“一个 sequence 对应一个线性 block table”升级为“一个 sequence 可以由多个带语义的 block group 组合而成”，同时保留旧路径兼容性，并将调度估算和 forward metadata 全链路改造成组合感知。
