Technical Design · DCP · GLM-5.2

Decode Context Parallel

xLLM DCP 完整设计方案：在不复制完整主 KV Cache 的前提下，将 Decode Attention 的上下文维度分布到多个设备 rank，并通过 Query 聚合、局部 Attention、 Head 重排和 LSE 合并恢复与单卡语义一致的输出；在 PD 分离部署中， 通过多源 Pack、Mooncake Staging 与 Decode Scatter 将 Prefill KV 转换为原生 DCP 布局。

01 / Executive summary

执行摘要

本次改动的核心不是新增一个独立 Attention Kernel，而是在现有 TP、Paged KV Cache、 MLA/DSA 和集合通信框架之间建立一条 Decode 专用的上下文并行数据通路。

1 / D

主 KV Cache 占用

DCP size 为 D 时，每个 rank 仅保存约 1/D 的主 MLA KV Token。

3 类

Decode 通信

Q AllGather、Output/LSE AllToAll，以及 DSA 全局候选 Top-K 汇聚。

Decode

阶段隔离

仅 Decode 进入 DCP；Prefill 与 Chunked Prefill 沿用原有路径。

Target

PD KV 重分片

设计支持 Prefill C + S 布局转换为 Decode D + I 布局；当前代码尚未实现。

一句话设计：每个 DCP rank 保存一段上下文 KV，却临时获得完整 Query Heads； 对本地上下文算出部分 Attention 后，按 TP Head 所有权交换部分结果，再用 Log-Sum-Exp 统计量进行精确合并，最终回到常规 TP 输出投影。
02 / Goals & boundaries

目标、动机与边界

问题

长上下文 Decode 时，主 KV Cache 随序列长度线性增长。TP 主要切分权重与 Head， 不能自然降低每个 rank 持有的上下文 KV 容量。

目标

把同一序列的 KV Token 分散到 DCP rank，在保持 Attention 数学语义和 TP 输出布局的同时， 降低单 rank KV Cache 压力，并复用现有 kv_split_size 的逻辑块机制。

当前范围

GLM-5.x/GLM-5.2 共用的 MLA 路径、Decode 阶段、DSA Lighting Indexer 和跨层 Top-K 共享，以及 PD 分离部署中的 KV Cache 布局协商与重分片目标方案。

设计内

  • DCP 配置贯穿 Master、Runtime Options 与 ParallelArgs
  • DCP ProcessGroup 建立与 rank 派生
  • 异步 AllGather / AllToAll 通信封装
  • 主 KV Cache 的 rank 归属和物理 Slot 重映射
  • MLA Decode 的局部计算、结果重排与 LSE 合并
  • 主 KV Cache 与 DSA Index Cache 使用一致的上下文分片
  • GLM-5.2 DSA Top-K 跨层复用
  • Prefill CP/KV Split 到 Decode DCP 的 PD KV 重分片协议

非目标 / 暂未覆盖

  • 不改变 Prefill CP 的算法与通信
  • 不在 Phase 1 支持 tp_size > dcp_size 的完整计算几何
  • 不改变 DSA Top-K 的选择语义和跨层复用关系
  • 不改变用户请求、调度器或 OpenAI API 语义
  • 不把本节 PD 重分片目标方案描述为当前代码已实现能力

04 / System architecture

总体架构

DCP 不是只存在于 Attention 层的局部优化。控制面负责把并行意图固化为 ParallelArgs 与 DCP 通信组；数据面负责 Slot、KV 和集合通信张量的移动； 模型面依据阶段选择执行路径，并以 AttentionMetadata、KVCache 和 DsaTopkTransfer 作为运行时接口。三者通过显式数据契约连接，不直接共享隐含布局。

06 / KV cache sharding

KV Cache 切分与 Slot 重映射

BlockManager 分配全局逻辑块，Worker 在 Decode 执行前决定当前 Token 由哪个 DCP rank 持有，并把逻辑 Slot 压缩到该 rank 的物理块空间；非本 rank Token 映射为 -1，由 Paged Cache 写入 Kernel 跳过。

映射公式

令 B = physical block_size, D = dcp_size, I = effective interleave_size
V = B × D                                  // 一个 DCP 逻辑块的 Token 数
v = position mod V                         // 逻辑块内偏移
owner = floor(v / I) mod D                 // Token 所属 DCP rank
local_offset = floor(v / (D × I)) × I + (v mod I)
logical_block_id = old_slot / V
physical_slot = logical_block_id × B + local_offset

mapped_slot = (owner == dcp_rank) ? physical_slot : -1

逻辑位置按 Token 轮转，进入各 rank 后再压缩成连续本地 Slot； rank 0 保存偶数位置，rank 1 保存奇数位置，局部序列长度约为 ceil(global_len / D)。

为什么逻辑块扩大 D 倍：调度器仍使用一个统一的逻辑 Block ID， 但该块在 DCP 组内由 D 个 rank 的物理块共同承载。这样可复用已有 BlockManager、 KV Split 和 PD 拓扑信息，而无需让调度层理解每个 Token 的实际设备归属。
07 / Decode data path

Decode 核心流程

STEP 01

本地 Q/KV 准备

按 TP shard 计算 Q；生成 MLA latent KV。主 KV 仅写入本 rank 拥有的 Slot。

STEP 02

Q AllGather

DCP 组聚合各 TP rank 的 Query Heads，使每个 rank 获得完整 Head 集合。

STEP 03

局部 Attention

每个 rank 用全量 Q 对本地 1/D KV 上下文调用 MLU batch_decode。

STEP 04

O/LSE AllToAll

按 Head 维把每个 KV shard 的 Partial Output 和 LSE 发给对应 TP Head owner。

STEP 05

稳定合并与投影

逐 KV shard 合并 O/LSE，恢复全上下文 Softmax 结果，再执行常规输出投影。

阶段代表性 Tensor 形状布局变化
本地 Q [T, Htp, Dh] 当前 TP rank 持有自己的 Query Heads。
Q AllGather 后 [D, T, Htp, Dh] → [T, D×Htp, Dh] Phase 1 中 D×Htp = Hfull。
局部 Decode Oᵣ: [T, 1, Hfull, L]
LSEᵣ: [T, Hfull, 1]		 r 表示 KV context shard；每个 rank 计算所有 Head。
AllToAll 后 [D, T, 1, Htp, L]
[D, T, Htp, 1]		 当前 rank 收到所有 KV shard 对自身 TP Heads 的局部结果。
合并后 [T, 1, Htp, L] 恢复当前 TP Head shard 的全上下文 Attention 输出。

LSE 合并为何是精确的

对于上下文分片 r，Kernel 返回局部归一化输出 Oᵣ 和 LSEᵣ = log Σ exp(score)。合并两个分片 a、b 时：

m = max(LSEₐ, LSEᵦ)
Z = exp(LSEₐ - m) + exp(LSEᵦ - m)
O = (exp(LSEₐ - m) × Oₐ + exp(LSEᵦ - m) × Oᵦ) / Z
LSE = m + log(Z)

反复调用 MLU update_out_and_lse 即可在不收集原始 Attention Score 的情况下，数值稳定地得到与全上下文 Softmax 等价的结果。

08 / DSA integration

DSA Index Cache 与 Top-K 共享

完整 DCP 方案中，主 MLA KV Cache 与 DSA Index Cache 都按照同一套 Token owner 和本地 Slot 规则切分。每个 rank 仅保存本地上下文的 KV 与 Index，Indexer 先产生本地候选， 再通过 DCP 集合通信恢复全局 Top-K 语义。当前分支采用的 Index 全副本与 Slot 恢复只是过渡实现， 不是最终设计。

Sharded

主 MLA KV Cache

继续使用 Worker 生成的 masked Slot Mapping；非本 rank Token 为 -1， 每个 rank 只写入自身拥有的 Token。

Sharded

DSA Index Cache

使用与主 KV 相同的 masked Slot Mapping，只写入本 rank Token 的索引向量。 Indexer 参数可按原模型并行方式放置，但历史 Index 数据不再全量复制。

分片 Indexer 与全局 Top-K

01

同源 Slot 分片

主 KV 与 Index Cache 共用 owner、logical block 和 local slot 映射，生命周期保持一致。

02

本地打分

每个 rank 的 Indexer 只扫描本地 1/D Index Cache，计算本地候选分数。

03

Local Top-K

选出本地 K 个候选，携带 score、全局 token/block id 和 owner rank。

04

候选 AllGather

聚合 D×K 个候选并执行 Global Top-K；所有 rank 得到同一组全局选择。

05

本地稀疏表

按 owner 过滤全局结果，把全局 id 转为本地物理 block id，生成 rank-local block table。

通信量：不收集完整 Index Cache，只交换每个 rank 的 K 个候选。 只要每个 rank 保留至少全局目标数量 K 的本地候选，全局 Top-K 必然包含在候选并集中。 对相同分数需要规定稳定的二级排序键（例如全局 token id），保证各 rank 选择完全一致。

09 / Disaggregated PD target design

PD 分离下的 DCP KV 重分片

Prefill 同时使用 cp_size=C 与 kv_split_size=S， Decode 使用 dcp_size=D 与 interleave_size=I。 两端的 KV owner、Block 几何和物理 Slot 可能不同，因此 PD 传输不能继续假定 “源物理块与目标物理块一一对应”，而要以 global token 为稳定坐标完成布局转换。

三个正交维度

Prefill 计算拓扑 cp_size = C

决定 Prefill Token 如何参与计算并行，不应被传输层当作 KV 唯一分片数。

Prefill Cache 布局 kv_split_size = S

决定 KV 的唯一物理 owner。建议要求 S | C； 当 C > S 时，其余 CP rank 可能持有副本，传输时只能选择一组唯一 writer。

Decode Cache 布局 dcp_size = D interleave_size = I

共同决定 global token 应进入哪个 Decode rank，以及在该 rank 上的 local slot。

KV Shard 与 KV Block 的关系

KV shard 是所有权概念，不是一个新的 Block 类型。 它描述某个 rank 在整条序列、所有逻辑 KV Block 中负责哪些 Token； KV Block 则是 BlockManager 与 Paged Cache 实际分配、寻址和传输的空间单位。

Logical KV Block V = B × S

调度器看到的连续 global-token 区间。示例中 B=4, S=2，一个逻辑块覆盖 pos 0..7。

→
S 个 KV Shard

shard 0 和 shard 1 分担该逻辑块。 同一个 shard 还会继续拥有后续 logical block 中相同规则的切片。

→
Rank-local Physical KV Blocks S × B tokens

每个 shard rank 为该逻辑块提供一个 B-Token 物理块； 物理 block id 可不同，global token mapping 才是稳定关系。

概念粒度生命周期与关系
KV Shard rank / ownership 一个 shard 贯穿请求的多个 KV blocks 和多个 Layer；回答“哪些 Token 由哪张卡保存”。
Logical KV Block B × S global tokens 调度与序列 Block Table 的逻辑单位，由同一 shard group 上的 S 个物理块共同承载。
Physical KV Block B local tokens 单张卡上 Paged Cache 的分配与传输单位；一个 shard 通常拥有很多 physical blocks。

“Gather → Send → Split”的准确含义：Gather 是每个 Prefill writer 从 rank-local physical KV blocks 取出有效 Token，并在逻辑上恢复 global-token 顺序； 上图用一条 canonical 行表达 Gather 结果。推荐实现可由多 writer 分布式完成本地 Pack 和 目标分桶，不要求任意单卡实际保存整条 canonical KV。Send 是多 writer 通过 Mooncake 写入各目标 DCP rank；Split 是 Decode 将 staging 条目 scatter 到最终 local slot。 推荐路径不要求 Prefill 集合通信产生一份完整 KV，也不设置单一汇聚节点。

推荐数据面：多源 Pack → Staging → Scatter

Prefill KV Writers

每个 unique KV shard 只读取本地 Cache；CP 副本不参与重复发送。

→
Rank-local Pack

按 cache role、layer chunk 和目标 dcp_rank 分桶，写入已注册的连续发送 staging。

→
Mooncake Multi-writer

每个 writer 写目标 rank staging 的独立 offset；Mooncake 只负责地址、长度和完成状态。

→
Decode Staging + Scatter

等待所需 writers 到齐，按 global token → local slot 计划写入最终 DCP Cache。

01 / NEGOTIATE

交换 Layout

P/D 交换 block size、S/D/I、dtype、cache roles、rank mapping 与 layout version。

02 / ALLOCATE

D 端预分配

各 DCP rank 分配最终 local blocks 和有界 ring staging，返回 session 与目标描述。

03 / PLAN

生成 Reshard Plan

以 global token 计算 source owner、destination owner 和 destination local slot。

04 / PACK + PUSH

本地打包直传

每个唯一 writer 按目标 rank 打包，并写入 writer-specific staging offset。

05 / BARRIER

多源完成屏障

Decode 按 request/layer/chunk/role/rank 跟踪所需 writer，禁止读取未完成数据。

06 / SCATTER

落入最终 Cache

设备侧 scatter 到 DCP local slot；event 完成后发布 KV-ready 并回收 staging。

12 / Performance model

性能与资源模型

显存收益

主 MLA KV Cache 与 DSA Index Cache 均从每 rank 完整持有降为约 1/D。模型权重和临时通信 Buffer 不随该比例下降。

计算变化

每个 rank 由“本地 Head × 全上下文”变为“全 Head × 1/D 上下文”。 在 TP=D 时主要 Attention FLOPs 近似守恒，但 Kernel 形状与访存行为改变。

通信成本

每层包含 1 次 Q AllGather + 2 次 AllToAll（O 和 LSE）。 Indexer 运行时再增加 1 次 D×K 候选 AllGather；Shared 层可复用本地稀疏状态， 省去 Indexer、Local Top-K 和候选通信。

通信近似载荷频率优化方向
Q AllGatherT × Htp × Dh / rank每个 DCP Attention 层与本地 KV 准备重叠；避免重复 contiguous。
O AllToAllHfull × T × L / rank每层与 LSE A2A 并发或融合通信。
LSE AllToAllHfull × T / rank，较小每层与 Output 打包成单次通信。
Top-K 候选 AllGatherK × (score + global id) / rank仅运行 Indexer 的层只交换候选而非完整 Index；Shared 层复用结果。
当前 recompute_dcp_cache_slots() 把 positions 和 slots 搬到 CPU 计算后再传回设备， 可能在小 Batch、逐 Token Decode 中放大同步开销。建议作为后续优化优先迁移到设备侧向量 Kernel， 或在调度阶段直接生成 rank-local Slot。
PD 重分片新增成本：推荐路径不传输重复 KV，P→D 网络有效载荷仍约为一份 Prompt KV；额外成本来自 Prefill 本地 pack、Decode staging scatter、chunk 元数据与完成屏障。 staging 峰值由 chunk_tokens × bytes_per_token × active_roles 控制， 不随完整上下文长度线性预留。分 chunk AllGather Fallback 还会增加 Prefill 组内通信和 KV 副本。
